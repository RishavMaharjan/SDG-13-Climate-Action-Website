Feedback Page

Technical Description

The feedback page is structured into three main parts inside <main>: a scrollable activity cards strip, a ratings and reviews section, and a user feedback form. The activity cards use display: flex; flex-wrap: nowrap; overflow-x: auto combined with scroll-snap-type: x mandatory to create a horizontally scrollable carousel, with each card fixed at flex: 0 0 300px to ensure consistent sizing. The ratings section employs a two-column CSS Grid with grid-template-columns: 1fr 2fr, placing an overall score panel alongside a responsive grid of individual review cards. The review cards use grid-template-columns: repeat(auto-fit, minmax(280px, 1fr)) to reflow across screen sizes. The feedback form is wrapped in a <fieldset> with a <legend> for clear semantic grouping. It includes a variety of input types: text, email, tel, date, datetime-local, and textarea. A <datalist> is used for the activity type field, offering suggestions while still allowing free text. A live character counter for the comment textarea is driven by a JavaScript input event listener. Form validation is handled entirely in JavaScript on the submit event, with inline error messages and a confirmation banner that auto-hides after five seconds using setTimeout.

Accessibility

Every form input has an explicit <label> element linked via matching for and id attributes, ensuring screen readers announce each field's purpose correctly. The form uses the novalidate attribute to suppress native browser popups in favour of custom, accessible error messages rendered in <span> elements adjacent to each field. The fieldset and legend combination groups the fields semantically. All card images carry descriptive alt attributes. The feedback form's submit button uses descriptive text rather than a generic label. The confirmation message uses scrollIntoView to direct keyboard and screen reader focus to the success notification.

Team Page

Technical Description

The team page uses a CSS Grid with grid-template-columns: repeat(auto-fit, minmax(280px, 1fr)) to display the four team member cards in a responsive layout that reflows from a four-column arrangement on wide screens down to a single column on mobile. Each card uses position: relative; overflow: hidden to contain a hidden detail panel. The detail panel is initially collapsed with max-height: 0; opacity: 0 and expands on :hover and :focus using a CSS transition on max-height and opacity, requiring no JavaScript. Member portrait images use object-fit: contain inside a fixed-size container so portraits of varying aspect ratios are displayed without cropping. A transform: scale(1.05) and border highlight are applied to the image on hover for a subtle interactive cue. The card layout uses display: flex; align-items: flex-start; gap: 1rem to place the thumbnail and member info side-by-side, collapsing to a vertical stack on small screens via a media query that sets flex-direction: column.

Accessibility

Each team member card has tabindex="0" applied so that keyboard users can focus the card and trigger the detail panel expansion via the :focus CSS state, mirroring the mouse hover interaction without requiring JavaScript event listeners. Every portrait image has a detailed alt attribute that names the person and describes their role. Heading hierarchy is maintained: <h3> is used for member names inside the cards, which sit below the page-level <h2>. The active navigation link includes class="active" to visually indicate the current page. The overall page structure uses proper landmark elements: <header>, <main>, and <footer>.

Content Page

Technical Description

The content page uses a two-column CSS Grid with grid-template-columns: 220px 1fr, placing a sticky side navigation <aside> alongside the main <article>. The aside uses position: sticky; top: 90px so it remains visible while the user scrolls through long content. The article is divided into semantic <section> elements, each with a unique id matching the corresponding anchor links in the side navigation. Heading hierarchy is strictly maintained: one <h1> for the page title, <h2> for topic sections, and <h3> for sub-points. Images use object-fit: cover with a fixed height, border-radius, and box-shadow for a consistent visual appearance. A fixed Go-to-Top button is implemented with position: fixed; bottom: 2rem; right: 2rem using a plain anchor link, requiring no JavaScript.

Accessibility

The side navigation uses an <aside> landmark element with an aria-label so screen readers can identify it separately from the main content area. All section headings have matching id attributes that serve as anchor targets, supporting smooth in-page scrolling. All images carry descriptive alt text explaining their content and relevance. The Go-to-Top button includes an aria-label attribute for screen reader users.

Challenges and Lessons Learned

Challenge 1: Building a Robust Feedback Form with Custom Validation
Implementing accessible, user-friendly form validation from scratch was more involved than expected. Native browser validation popups are not consistently styled or announced by screen readers, so I used the novalidate attribute and wrote custom JavaScript handlers to show/hide error messages dynamically in the DOM adjacent to each field. Ensuring error messages were reliably linked to their inputs required careful use of matching id and for attributes throughout the form.

Challenge 2: CSS-Only Interactive Team Cards
My initial plan used JavaScript event listeners to toggle the detail panel on each team card, but I wanted the interaction to degrade gracefully without scripting. I discovered that combining :hover and :focus CSS pseudo-classes with a max-height transition achieves the same reveal effect purely in CSS. The main difficulty was preventing layout jump when the panel expanded, solved by using a sufficiently large max-height value and tuning the transition duration.

Challenge 3: Horizontal Scroll Carousel Without a Library
Creating a smooth, snapping horizontal scroll for the activity cards without a third-party carousel library required combining overflow-x: auto, scroll-snap-type, and custom scrollbar styling. Ensuring the scrollbar was visible but unobtrusive across browsers took several iterations of testing ::webkit-scrollbar rules.

Lessons Learned
This project reinforced that accessibility and interactivity are not mutually exclusive. Many interactive effects can be achieved with pure CSS in a way that is inherently more accessible than JavaScript alternatives. I also learned the importance of early cross-browser testing, as scroll behaviour and custom scrollbars vary significantly between Chrome, Firefox, and Safari.
