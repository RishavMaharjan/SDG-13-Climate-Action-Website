Home Page

Technical Description

The home page is structured into four semantic sections inside <main>: a hero banner, a statistics strip, a mission cards section, and a content gallery. The hero uses CSS Grid with grid-template-columns: 1fr 1.35fr to position the text and image side by side, collapsing to a single column on smaller screens via a media query. The statistics strip uses grid-template-columns: repeat(3, 1fr) to display three equally spaced figures. Mission cards are laid out using a three-column grid and are each wrapped in an <article> element, which is the correct semantic choice since each card represents a self-contained piece of content. The gallery uses a two-column grid of anchor tiles, each with a hover lift and scale effect driven by CSS transition and transform. The page links to the shared style.css file and adds no page-specific embedded styles of its own.

Accessibility

All sections include either aria-label or aria-labelledby attributes to give screen readers meaningful context. The hero buttons use descriptive link text rather than generic labels. All images have detailed alt attributes describing their content and relevance. The navigation uses a <nav> landmark with aria-label="Main navigation" and the active page is marked with class="active" and aria-current="page".

Gallery Page

Technical Description

The gallery is built on a three-column CSS Grid of thumbnail cards. Each card uses position: relative on the container and position: absolute on the overlay element to create a caption that slides up on hover using transform: translateY() and opacity transitions. Hover and click events are handled entirely in JavaScript using addEventListener. Clicking a card calls openModal(id), which looks up an entry in the galleryData array and populates the modal's image, title, and description. The modal is a fixed-position <div> that can be closed via the close button, by clicking outside the modal box, or by pressing the Escape key. The modal also includes a customisation panel where users can change text colour and font family using inline style properties applied to the content area.

Accessibility

Each gallery card has role="listitem" and tabindex="0", with a keydown listener that opens the modal on Enter or Space, mirroring native button behaviour. The modal uses role="dialog", aria-modal="true", and aria-labelledby="modal-title" to follow ARIA dialog patterns. Focus is moved to the close button when the modal opens. Every image has a descriptive alt attribute.

Content Page

Technical Description

The content page uses a two-column CSS Grid with grid-template-columns: 220px 1fr, placing a sticky side navigation <aside> alongside the main <article>. The aside uses position: sticky; top: 90px so it remains visible as the user scrolls through long content. The article is divided into six <section> elements, each with a unique id matching the anchor links in the navigation. Heading hierarchy is strictly maintained: one <h1> for the page title, <h2> for topic sections, and <h3> for sub-points. Images use object-fit: cover with a fixed height, border-radius, and box-shadow for a consistent appearance. A fixed Go-to-Top button is implemented with position: fixed; bottom: 2rem; right: 2rem and a plain anchor link, requiring no JavaScript.

Accessibility

The side navigation uses an <aside> landmark element with an aria-label so screen readers can identify it separately from the main content. All section headings have matching id attributes that serve as anchor targets with smooth scrolling. All images have descriptive alt text. The Go-to-Top button has an aria-label attribute for screen reader users.

Challenges and Lessons Learned

Challenge 1: Creating a Reusable Template
The most significant challenge was designing the HTML template and global CSS to be flexible enough for four different page types, yet strict enough that teammates could not accidentally break shared structure. I resolved this by adding comments in both HTML and CSS clearly marking which sections must not be modified, and by using CSS custom properties (var()) so teammates could apply colours without needing to know the exact hex values.

Challenge 2: JavaScript Modal Accessibility
Making the gallery modal fully accessible, including keyboard navigation, was more complex than expected. I researched ARIA roles (role="dialog", aria-modal) and implemented focus management so the close button receives focus when the modal opens. I added an Escape key listener and a click-outside handler to offer multiple intuitive ways to close it.

Challenge 3: Responsive Layout Without CSS Conflicts
Ensuring page-specific styles did not conflict with the global CSS required careful naming. I resolved this by using specific class names for page-only elements (such as .hero and .content-layout) that cannot clash with global names.

Lessons Learned
This project showed me the value of building a design system before writing page content. The time spent on the shared template and CSS file saved significant effort overall and produced a more consistent result. I also learned that accessibility needs to be considered from the very start of development, not added at the end.
